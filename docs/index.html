01 — The problem

Two price cliffs swallow ~94% of the funnel

A thousand in-scope visitors start the private-doctor calculator. By the closing step only ~56 remain. The leaks aren't spread evenly — they're concentrated at the moment a price appears.

Where they leave

At step 4 (initial price) 66% of the surviving cohort drops. At step 7 (final price after the health questions) another 78% of who's left drops — often because the final price ticks up from the estimate with no clear explanation.

Survival math: 1,000 → ~340 → ~57 ≈ the ~5.6% baseline UNIQA reports. The coach's whole job is to widen those two bottlenecks without annoying the people who'd have stayed anyway.

02 — How it works

A two-stage system, one source of truth

Edit a persona .md and the simulated people change. Edit personas.js and the coach's config changes. Nothing else hard-codes persona facts — so the demo can never drift from the engine being judged.

📋 source of truth franz.md · judith.md · peter.md  → the characters the LLM role-plays personas.js  → segmentation + funnel facts (50/30/20 mix · 66/78 cliffs)
Stage 1 · GPU — Leonardo A100 · vLLM

Act out real journeys

A real LLM role-plays each persona through the funnel — 1 call = 1 full journey — recording behavioural signals in character.

  • Franz bails at the final-price surprise
  • Judith hesitates at the initial price
  • Peter is overwhelmed early
  • Outputs signal ranges: dwell, hesitation, back-clicks, competitor-tab
Stage 2 · CPU — runs anywhere (~1 min)

Calibrate, train, evaluate

A probabilistic funnel is calibrated to UNIQA's real numbers, emits labelled per-step rows, and trains a small classifier — the coach's brain.

  • funnel.py reproduces 66% / 78% / 5.6%
  • ~110k labelled rows → HistGradientBoosting
  • Common random numbers → perfectly paired before/after
  • evaluate.py → the three judged dimensions

04 — The coach

Detect when, decide how

The trained classifier outputs P(abandon at this step). The coach blends that with transparent hard rules, infers a coarse segment from behaviour (no privileged knowledge of the true persona), and fires the intervention documented as effective for that segment.

DETECT — when? coach/coach.py

  • Classifier risk. P(abandon) from the gradient-boosting brain, on signals observable at the moment.
  • Hard rules. Final-price jump · ≥2 back-clicks · competitor tab · long dwell.
  • Dropoff vigilance. The firing threshold is lowered at the step where the inferred segment is documented to drop off.

DECIDE — how? grounded in personas.js

  • Infer segment. online / hybrid / service, from the signal pattern alone.
  • Pick best intervention. from that segment's documented best_coach_interventions.
  • Resolve vs the true persona. a mis-targeted nudge earns no uplift — or backfires — which the intervention-quality metric captures.

Every fire is logged with its trigger reason

07 — Honesty & scope

What's real, what's synthetic, disclosed up front

The full evaluation, assumptions and disclosures live in REPORT.md. The highlights:

✅ Out-of-scope = correct exit

Hospital path, "other persons", and the advisory-only tariffs (Opt.Plus / Premium) force a clean advisor route. It's counted as advisor_routed and reported separately — never as an online conversion. The coach routes them, it doesn't coach them.

⚖️ The conversion-definition conflict

The brief contradicts itself (contract spec: online only; persona matrix: handoffs also count). We use the stricter contract definition as the headline and track advisor routes separately — we did not silently pick the more favourable number.

🔁 Reproducible

Identical seeds for baseline vs coach (common random numbers), verified not assumed. Training seed 11, evaluation seed 99 (disjoint). Re-runs end-to-end on CPU in ~1 min via ./run.sh.

🧪 Synthetic, documented

No real clickstream — the funnel is calibrated to UNIQA's published aggregates (66/78/5.6), and response efficacies (including the documented backfires) are synthetic and stated. The LLM run was a smoke-test-sized validation of the pipeline.

🧠 Not an LLM wrapper

Detection is a trained HistGradientBoostingClassifier blended with transparent rules. The optional Phi-3 chat is not required — without it replies are instant and doc-grounded, and the decision logic is independent of any LLM call.

🔒 Clean submission

MIT licensed, public repo, no API keys or tokens in the repo or its git history. README + REPORT + requirements present; runs from a clean checkout.

08 — Run it yourself

One launcher, zero setup pain

run.sh builds a local pip venv (no conda solve) and runs anything. The web demo has two tabs: a live click-through coach, and the real-engine evidence view.

# 1 · clone & launch the web demo git clone https://github.com/ValGSgit/ZeroOneHack-PromptCorn-Sub.git cd ZeroOneHack-PromptCorn-Sub ./run.sh app # → http://localhost:9696 (two tabs: live coach + evidence)
# 2 · reproduce the analysis (CPU, ~1 min) ./run.sh calibrate # verify 5.6% / 66% / 78% ./run.sh eval --no-plots # the three dimensions ./run.sh demo --compare judith --seed 17 # → terminal side-by-side: routed → CONVERTED
